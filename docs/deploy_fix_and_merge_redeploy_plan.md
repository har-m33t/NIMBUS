# NIMBUS Deploy Fixes, Merge Strategy, and Redeploy Playbook

## 1) Current Status

- Backend SAM deploy now succeeds in us-west-2 when deployed with Mediasoup disabled.
- The stack moved past early change-set validation failures and completed update after template hardening.
- Main blocker that was resolved last: `SessionsTable` replacement restrictions for a custom-named DynamoDB table.

## 2) What Failed and Why

### A) Early validation failures (pre-update)

CloudFormation change set creation repeatedly failed with:

- `AWS::EarlyValidation::ResourceExistenceCheck`

Primary contributors were:

- Logical-ID drift for connect/disconnect Lambda resources (case mismatch such as `Ws...` vs `WS...`) creating Add/Remove conflicts for already-existing physical resources.
- Explicit management of pre-existing CloudWatch log groups for connect/disconnect Lambdas.
- Optional Mediasoup resources still participating in validation flow during standard backend-only deploys.

### B) Update rollback during execution

Once early validation was fixed, update failed on:

- `SessionsTable` requiring replacement while being custom named (`NIMBUS_PROD_Sessions`), which CloudFormation blocks for replacement in-place.

Error observed:

- `CloudFormation cannot update a stack when a custom-named resource requires replacing. Rename NIMBUS_PROD_Sessions and update the stack again.`

## 3) Exact Fixes Applied

## Backend template and deploy config

### 3.1 Mediasoup hard-disable control for backend-only deploys

File: backend/template.yaml

- Added parameter `DeployMediasoupEnabled` with default `false`.
- Added conditions:
	- `MediasoupExplicitlyEnabled`
	- `HasMediasoupVpcId`
	- `HasMediasoupSubnetId`
	- `HasMediasoupKeyPairName`
	- `DeployMediasoup` (AND of all above)
- Applied `Condition: DeployMediasoup` to all Mediasoup EC2-related resources and output.

Why:

- Prevents accidental Mediasoup validation/deployment in normal backend-only releases.

### 3.2 Restored stable logical IDs for connect/disconnect Lambda resources

File: backend/template.yaml

- Reverted logical IDs and references to `WSConnect...` and `WSDisconnect...` forms.
- Updated API Gateway integration URIs to reference restored IDs.

Why:

- Eliminates Add/Remove churn for physical resources that already exist in stack state.

### 3.3 Removed explicit connect/disconnect log-group resources

File: backend/template.yaml

- Removed `WsConnectLogGroup` and `WsDisconnectLogGroup` resources.

Why:

- These often already exist from Lambda runtime creation and can fail existence checks during updates.

### 3.4 Allowed Sessions table replacement migration

File: backend/template.yaml

- Removed explicit `TableName` from `SessionsTable`.

Why:

- Allows CloudFormation to replace the table when schema evolution requires replacement.

Important effect:

- New physical table name is auto-generated by CloudFormation.
- Old custom-named table is retained due to `Retain` policies.

### 3.5 Fixed CloudFormation pattern issue in SecurityGroup description

File: backend/template.yaml

- Replaced Unicode em-dash with ASCII hyphen in SG description.

Why:

- Avoids strict pattern validation issues in certain CloudFormation checks.

### 3.6 SAM default deploy override safety

File: backend/samconfig.toml

- Added deploy default:
	- `parameter_overrides = "DeployMediasoupEnabled=false"`

Why:

- Keeps normal `sam deploy` behavior safe and consistent.

### 3.7 Packaging cleanup (Lambda artifact size and determinism)

Files:

- backend/requirements.txt
- backend/requirements-dev.txt

Changes:

- Removed test/development dependencies from runtime requirements.
- Kept `moto` in dev requirements only.

Why:

- Reduces deployment package size and avoids shipping test libs in Lambda runtime.

## 4) Validation Performed

- `sam validate` passed after each template iteration.
- `sam build --cached` succeeded after each template iteration.
- Final `sam deploy` completed successfully with:
	- `DeployMediasoupEnabled=false`

## 5) What Needs To Happen Before Merging

## 5.1 Backend merge readiness checklist

- Confirm no remaining change-set Add/Remove churn for existing `WSConnect/WSDisconnect` function resources.
- Confirm stack outputs still provide required keys (`WebSocketURL`, `SageMakerEndpointName`, buckets, table names).
- Confirm application code reads table names from environment, not hard-coded names.
- Confirm no one relies on old fixed physical `NIMBUS_PROD_Sessions` table name.

## 5.2 Required one-time data migration decision (important)

Because `SessionsTable` now allows replacement and old table is retained:

- Decide whether to migrate existing session/caption/rate-limit records from retained old table to new table.
- If no migration is needed for transient session data, document that explicitly.

Recommended:

- Treat session/gloss/rate-limit data as ephemeral and skip migration unless analytics/history requires it.

## 5.3 Frontend merge readiness

Frontend changes in this workspace are substantial and should be merged deliberately:

- WebSocket client implementation
- MediaPipe extraction and face-crop logic
- 10 FPS frame pacing
- `.env` usage

Before merge:

- Ensure frontend references stack output key `WebSocketURL`.
- Ensure `.env` template points to the correct deployed stage and region.
- Verify message schema matches PROTOCOLS (INFER payload, face crop cadence, session/room fields).

## 6) Member 3 Integration (ML/SageMaker) After Their Merge

## 6.1 Expected handoff contracts

- Artifact uploaded to `ModelArtifactsBucketName` output path.
- Endpoint deployed as `SageMakerEndpointName` output value, or parameter overridden intentionally.
- Endpoint state `InService` before end-to-end tests.

## 6.2 Backend compatibility checks

- `ProcessFrame` and warm endpoint functions still point to the intended endpoint name via env var.
- IAM remains sufficient for `sagemaker:InvokeEndpoint` on target endpoint.
- Failure fallback behavior remains intact when endpoint is unavailable.

## 7) Recommended Merge Order

1. Merge backend infrastructure fixes first (this set).
2. Deploy backend stack once from merged backend branch.
3. Merge Member 3 endpoint/model updates.
4. Redeploy backend with final `SageMakerEndpointName` if changed.
5. Merge frontend branch and wire env to current stack outputs.
6. Run end-to-end validation across capture -> inference -> caption -> tts.

## 8) Redeploy Playbook After Backend + Member 3 + Frontend Merge

## 8.1 Preflight

- Confirm temporary AWS credentials are valid (`aws sts get-caller-identity`).
- Set `AWS_PAGER=""`.
- Pull latest merged main and ensure clean local branch/worktree.

## 8.2 Backend deploy

Use backend-only deploy default:

- `sam build --cached`
- `sam deploy --stack-name nimbus-prod --region us-west-2 --capabilities CAPABILITY_IAM CAPABILITY_NAMED_IAM --resolve-s3 --no-confirm-changeset --no-fail-on-empty-changeset --parameter-overrides DeployMediasoupEnabled=false`

If Member 3 changed endpoint name:

- Add `SageMakerEndpointName=<new-endpoint-name>` in `--parameter-overrides`.

## 8.3 Frontend env wiring

Read outputs and set frontend env:

- `WebSocketURL` -> `NIMBUS_WS_URL`
- any additional shared values as needed

## 8.4 Post-deploy verification

- WebSocket connect/disconnect works.
- INFER route accepts 10 FPS payloads.
- GLOSS/EMOTION/CAPTION/ERROR events received as expected.
- Polly audio URLs return valid content.
- Session writes/reads occur in the new Sessions table.

## 9) Operational Risks and Mitigations

- Risk: retained old Sessions table causes confusion.
	- Mitigation: document active table from stack output and deprecate old table clearly.

- Risk: future logical ID renames trigger resource replacement churn.
	- Mitigation: treat logical IDs as stable API; do not rename unless planned migration.

- Risk: credentials expire mid-deploy and produce misleading errors.
	- Mitigation: refresh credentials immediately before deploy and verify with STS.
